# Безопасность при работе с AI-агентами

Этот документ — про то, как ограничить агента, который работает в терминале с этой репой.
Правила для самого агента лежат в [`AGENTS.md`](../AGENTS.md); здесь — как владелец
настраивает окружение, чтобы правила нельзя было просто проигнорировать.

## Ключевой тезис: branch protection ≠ граница безопасности

Это два разных уровня, их легко перепутать:

| | Что защищает | От чего НЕ защищает |
|---|---|---|
| **Branch protection** (на сервере GitHub) | от случайного/некорректного коммита в `main`; даёт обязательное ревью диффа | от агента с твоим токеном — он снимет защиту сам |
| **pre-push хук** (локально) | от случайного `git push` в `main` | обходится флагом `--no-verify` |
| **Урезанный токен + песочница** | реально ограничивает права | — |

Если у агента есть залогиненный `gh` или твой personal access token с правами админа,
он действует **с правами владельца репозитория**. То есть может:

```bash
gh api --method DELETE repos/Vadim-B52/MN/branches/main/protection   # снять защиту
gh repo edit --visibility private                                     # сменить настройки
gh repo delete Vadim-B52/MN                                           # удалить репу
```

Branch protection ловит **ошибки**, а не злонамеренное или сбойное поведение агента
с полными ключами. Настоящее ограничение — это права токена и изоляция окружения.

## Что реально ограничивает агента (по возрастанию надёжности)

### Уровень 1 — урезанный fine-grained токен

Не давать агенту свой `gh`-логин. Вместо этого — токен только на эту репу,
без прав администрирования.

**Как создать:**

1. Открыть https://github.com/settings/personal-access-tokens/new
2. **Token name:** `mn-agent`
3. **Expiration:** 90 дней (короче — лучше; продлевать вручную)
4. **Repository access** → *Only select repositories* → `Vadim-B52/MN`
5. **Permissions** → Repository permissions:

   | Permission | Значение | Зачем |
   |---|---|---|
   | Contents | **Read and write** | читать код, пушить ветки |
   | Pull requests | **Read and write** | открывать PR |
   | Metadata | **Read-only** | обязательное, ставится само |
   | Administration | **No access** | ← критично: без этого агент снимет branch protection |
   | всё остальное | **No access** | |

6. **Generate token** → скопировать `github_pat_...` (показывается один раз)

**Использование:**

```bash
export GH_TOKEN=github_pat_xxxxxxxx
gh auth status
```

**Проверка, что токен действительно урезан** — команда ниже должна упасть с `403`:

```bash
GH_TOKEN=github_pat_xxxx gh api --method DELETE \
  repos/Vadim-B52/MN/branches/main/protection
```

Если она отработала успешно — у токена есть Administration, переделать.

### Уровень 2 — отдельный аккаунт для агента

Завести GitHub-аккаунт `mn-agent`, добавить его в репу с ролью **Write** (не Admin).
Агент работает под его токеном. Тогда можно включить
`required_approving_review_count: 1` — агент физически не сможет заапрувить
собственный PR, а ты апрувишь под своим аккаунтом.

Это единственный способ получить настоящий обязательный апрув
(см. оговорку про self-approve в [`branch-protection.md`](./branch-protection.md)).

### Уровень 3 — песочница

Урезанный токен не спасает, если агент работает в твоём основном окружении:
там лежат `~/.ssh`, `~/.config/gh`, `.env` других проектов — он может их прочитать
или использовать.

Варианты по возрастанию изоляции:

1. **Отдельный OS-пользователь** — просто, но общая файловая система.
2. **Docker / dev container** — репа примонтирована, в env только нужный `GH_TOKEN`,
   доступа к домашней директории хоста нет. Оптимальный баланс.
3. **Отдельная VM** — максимальная изоляция, дороже по ресурсам.

Разбор этой темы с практикой (`.devcontainer/`) запланирован —
см. [`learning-agenda.md`](./learning-agenda.md), пункт 2.

## Чек-лист перед тем, как пускать агента

- [ ] Создан fine-grained токен `mn-agent` с Administration = No access
- [ ] Проверено, что `DELETE .../protection` под этим токеном падает с 403
- [ ] Агент запускается с `GH_TOKEN`, а не с твоим `gh auth login`
- [ ] В окружении агента нет доступа к `~/.ssh` и `~/.config/gh`
- [ ] `pnpm install` выполнен — значит `.husky/pre-push` активен
- [ ] Агент прочитал `AGENTS.md`
