# Процесс работы

Как большие куски превращаются в то, что можно сделать за одну сессию.

## Четыре уровня

```
Цель          зачем · итог, а не действие · 1–2 квартала · docs/goals.md
 └─ Проект    крупный кусок · недели · один файл в docs/projects/
     └─ Задача      день–неделя · раздел внутри файла проекта
         └─ Минизадача   одна сессия (≈1–3 часа) · чекбокс внутри задачи
```

Правило: **минизадача — это то, что реально закрывается за один заход.**
Если не закрывается — она была задачей, разбей дальше.

## Файлы и их роли

| Файл | Роль | Горизонт |
|---|---|---|
| `docs/goals.md` | Цели: зачем мы вообще | кварталы |
| `docs/projects/` | Проекты: по файлу на каждый, внутри задачи и минизадачи | недели |
| `docs/next-session-agenda.md` | Что берём в следующий заход | одна сессия |
| `docs/backlog.md` | Инбокс: всё, что прилетело, но пока не разобрано | — |
| `docs/roadmap.md` | Временной срез: milestones | месяцы |

Отличие бэклога от проектов: **бэклог — свалка входящего**, проекты — уже
структурированная работа. Мысль сначала падает в бэклог, потом либо становится
задачей внутри проекта, либо новым проектом, либо удаляется.

## Цикл одной сессии

**Открытие (2 минуты):**
1. Открыть `docs/next-session-agenda.md`.
2. Решить по каждому пункту: **делаю** / **в бэклог** / **удалить**.
3. Что делаем — держим в голове/в задачнике агента.

**Работа.** Отмечаем минизадачи `[x]` по мере закрытия.

**Закрытие (5 минут):**
1. Проставить `[x]` в файле проекта.
2. Обновить статус проекта, если сменился.
3. **Заполнить `next-session-agenda.md` на следующий раз** — пока контекст свежий.
   Это главный шаг: без него следующая сессия начнётся с «а что я делал?».
4. Значимое решение → `ai/context/decisions-log.md` (или `docs/adr/` если развёрнуто).

Агент умеет вести этот цикл сам — см. `ai/skills/session-workflow/SKILL.md`.

## Статусы

`🔜 план` · `🚧 в работе` · `✅ готово` · `❄️ заморожен` · `❌ отменён`

Держи **не больше 1–2 проектов в статусе 🚧**. Остальное — 🔜 или ❄️.
Это главная защита от расползания в соло-проекте.

## Идентификаторы

- Проект: `P01`, `P02`, … — файл `docs/projects/P01-краткое-имя.md`
- Задача: `P01-T1`, `P01-T2` — раздел внутри файла проекта
- Минизадача: просто чекбокс, без номера

Ссылаться в коммитах и PR можно так: `feat(P02-T1): ...`

## Почему markdown, а не GitHub Issues

Источник правды — файлы в репе: их читает AI-агент вместе с кодом, они версионируются,
работают офлайн и не требуют переключения контекста. GitHub Issues имеет смысл
подключить позже — когда появятся другие люди или внешние баг-репорты. Тогда
Issues = оперативка, а `docs/` остаётся уровнем целей и проектов.

## Чего НЕ делать

- Не заводить проект под каждую мелочь — мелочь живёт в бэклоге или как минизадача.
- Не писать план на 3 месяца вперёд по минизадачам — детализируй только ближайшее.
- Не держать 5 проектов «в работе» одновременно.
- Не оставлять `next-session-agenda.md` пустым в конце сессии.
